# Skill: 任务拆解

## 触发条件

收到用户需求时使用，将需求拆解为可分配的子任务。

## 拆解框架

### 第 1 步: 需求分类

- 新功能开发（走完整 7 阶段）
- Bug 修复（跳过算法设计，直奔测试+修复）
- 文档更新（仅 Lens + Atlas）
- 性能优化（Euler 分析 + Forge 优化 + Sentinel 验证）

### 第 2 步: 依赖分析

```
哪些子任务可以并行？
哪些必须串行（有依赖）？
关键路径在哪？
```

### 第 3 步: 分配矩阵

| 子任务 | 负责人 | 依赖           | 优先级   | 预估复杂度 |
| ------ | ------ | -------------- | -------- | ---------- |
| [任务] | [成员] | [依赖的子任务] | P0/P1/P2 | 低/中/高   |

### 第 4 步: 可分解性判断

满足以下任一条件 → 可分解:

- 步骤 ≥ 3 个
- 涉及 ≥ 2 个成员
- 子任务之间可并行

## 工作流阶段速查

| 阶段       | 主要负责人           | 产出物          |
| ---------- | -------------------- | --------------- |
| 1 需求分析 | Marshall（Leader）   | 子任务清单      |
| 2 算法设计 | Euler（算法设计师）  | 算法方案+伪代码 |
| 3 代码开发 | Forge（代码开发）    | 源代码          |
| 4 代码测试 | Sentinel（代码测试） | 测试报告        |
| 5 代码分析 | Lens（代码分析）     | 分析报告        |
| 6 文档编写 | Atlas（文档工程师）  | 软件说明书      |
| 7 汇总交付 | Marshall（Leader）   | 最终交付物      |
